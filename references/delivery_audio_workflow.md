# 视听拉片、音频分析与交付升级规则

## 触发场景

当用户提出以下任一需求时，必须读取并执行本文件：

- “完整看一遍视频”“逐秒拉片”“每一秒变化”“图文拉片”。
- “专业导演分析”“导演意图”“为什么这么拍”“从导演角度分析”“多角度分析拍摄意图”。
- “风格圣经”“风格迁移”“参考风格”“给其他 AI 当知识库”“场景图提示词”“视频提示词”。
- “分析音频”“音乐、音效、人物、对白都要写清楚”。
- “保存到某个输出目录”“上传飞书”“转 Word”“图片嵌入 MD / DOCX”。
- 用户要求导演级、分镜师级、AI 视觉专家级复盘，且视频素材可被本地处理。

## 交付前先定输出规格

开始长篇分析前，先确定或自动记录以下规格：

| 项目 | 规则 |
|---|---|
| 输出目录 | 如果用户指定目录，报告、抽帧、音频分析、Word/PDF 预览都放在该目录或其子目录。 |
| 报告格式 | 默认先产出 Markdown；如果用户要飞书上传或嵌图交付，优先追加 DOCX 嵌图版。 |
| 图片策略 | Markdown 可使用相对图片路径；DOCX 必须真正嵌入图片，不能只放本地链接。 |
| 基础拉片报告 | 完整本地视频拉片必须先有“帧/自适应详细拉片报告”；它是后续专业导演分析和风格圣经的证据基础。 |
| 专业导演分析 | 完整本地视频拉片默认额外产出独立“专业导演分析”Markdown 草稿和嵌图 DOCX；它基于基础拉片报告，专门回答为什么这样拍、导演意图、观众心理和可学习方法。触发边界以 SKILL.md "default three-deliverable package" 为准。 |
| 风格圣经 | 完整本地视频拉片默认额外产出独立“源片风格圣经”Markdown 草稿和嵌图 DOCX；它用于其他 AI 学习风格，不要和主拉片报告混成一个 Word。触发边界以 SKILL.md "default three-deliverable package" 为准（快速看一下/局部段落/纯截图剧本任务默认不触发）。 |
| 音频策略 | 用户提到音乐、音效、对白、人物声音时，必须做音频提取、ASR 或字幕对照、声音结构分析。 |
| 证据等级 | 画面、字幕、ASR、波形为 E1；由多项证据推断出的情绪/主题为 E2；无法确认则标 X。 |
| 归档命名 | 默认使用 `08拉片输出/YYYYMMDD/片名_任务名/`，例如 `08拉片输出/20260609/万物生_问心/`。 |

如果视频文件后续被移动或删除，但抽帧、ASR、波形等证据仍在，报告中要明确“源视频当前不可复验”，并说明本次分析依赖哪些已落盘证据。

如果用户要求“继续”“看看哪些已经完成”“补 Word/PDF/QA”等续跑任务，先运行项目状态扫描，不要凭目录肉眼判断：

```bash
python scripts/lapian_delivery_status.py --project-dir 08拉片输出/YYYYMMDD/片名_导演级拉片
```

以输出里的 `done` 和 `next_steps` 作为续跑清单。已完成且未过期的环节不重复生成；如果 Markdown 主报告被改动，则 DOCX、PDF、QA、manifest 都视为需要重新收口。

## 文件夹命名与归档规范

### 1. 默认根目录

新拉片任务默认归档到：

```text
08拉片输出/
```

如果项目里同时存在 `拉片输出/`、`08拉片输出/` 或带横杠的旧目录，优先遵循用户本次明确指定的目录；用户未指定时，新任务统一使用 `08拉片输出/`，不要再随意新增相近名称。

### 2. 日期批次目录

日期目录统一使用 8 位数字：

```text
YYYYMMDD
```

示例：

```text
20260609
```

表示 2026 年 6 月 9 日开始执行或主要完成的拉片批次。若任务跨日，目录仍用任务开始日期，并在 `manifest.json` 或交付审计里记录完成日期。

### 3. 单片项目目录

日期目录下按片名和任务名建项目目录：

```text
08拉片输出/20260609/万物生_问心/
08拉片输出/20260609/Nexus_导演级逐秒拉片/
```

命名规则：

- 用中文可读片名或用户给定标题。
- 片名与任务名之间用 `_`。
- 删除 Windows 路径非法字符：`< > : " / \ | ? *`。
- 不使用随机 ID、下载 hash、纯英文占位名作为最终目录名。
- 同一天同一片名多次输出时，加任务后缀或版本号，例如 `万物生_问心_v02`、`万物生_问心_音频复核版`。

### 4. 项目内子目录结构

推荐结构：

```text
08拉片输出/20260609/片名_任务名/
  00_源信息/
  01_逐秒抽帧/
  02_音频分析/
  03_MD报告/
  04_飞书Word交付/
  05_PDF预览/
  06_QA审计/
  manifest.json
```

各目录职责：

| 子目录 | 放什么 |
|---|---|
| `00_源信息/` | `ffprobe` 信息、视频元数据、用户需求摘要、素材可用性说明。 |
| `01_逐秒抽帧/` | 原始逐秒帧、关键帧、contact sheet、抽帧 manifest、镜头切分 JSON（场景检测剪辑点与镜头表）。 |
| `02_音频分析/` | WAV、ASR JSON/TXT/JSONL、音频能量 CSV、对白校对稿。 |
| `03_MD报告/` | Markdown 主报告、帧/自适应详细拉片报告、专业导演分析、源片风格圣经、分段草稿。 |
| `04_飞书Word交付/` | DOCX 嵌图版、专业导演分析嵌图版、源片风格圣经嵌图版、Word 图片压缩缓存、飞书上传用版本。 |
| `05_PDF预览/` | PDF 预览、PDF 渲染截图、预览 contact sheet。 |
| `06_QA审计/` | 图片引用检查、缺失文件检查、DOCX 嵌图检查、最终审计记录。 |

如果是小任务，可以减少子目录，但至少保留：抽帧、音频分析、报告、交付文件四类不要混放。

### 5. 文件命名规则

主文件使用：

```text
片名_报告类型_v01.md
片名_报告类型_嵌图版_v01.docx
片名_报告类型_QA预览_v01.pdf
片名_专业导演分析_v01.md
片名_专业导演分析_嵌图版_v01.docx
片名_源片风格圣经_v01.md
片名_源片风格圣经_嵌图版_v01.docx
```

示例：

```text
万物生_问心_导演级视听逐秒拉片报告_v01.md
万物生_问心_导演级视听逐秒拉片报告_嵌图版_v01.docx
万物生_问心_导演级视听逐秒拉片报告_QA预览_v01.pdf
万物生_问心_专业导演分析_v01.md
万物生_问心_专业导演分析_嵌图版_v01.docx
万物生_问心_源片风格圣经_v01.md
万物生_问心_源片风格圣经_嵌图版_v01.docx
```

不要覆盖已有版本。需要重新生成时递增 `v02`、`v03`，或使用清晰语义后缀，例如 `_音频修订版`、`_飞书交付版`。

### 6. manifest 记录

每个项目目录建议保留一个 `manifest.json`，记录：

- 源视频路径、文件名、大小、时长、分辨率、fps。
- 拉片开始日期、完成日期、负责人或执行模型。
- 抽帧间隔、抽帧数量、最后一帧时间码。
- ASR 模型、字幕来源、音频分析文件。
- 主报告、DOCX、PDF、QA 文件路径。
- 专业导演分析 Markdown、嵌图 DOCX、选用代表帧数量和覆盖场景数量。
- 源片风格圣经 Markdown、嵌图 DOCX、选用关键帧数量和风格资产卡数量。
- 当前限制，例如源视频缺失、ASR 不可靠、无法导出 PDF 等。

最终交付前用脚本回写交付路径和完成度，避免 manifest 只记录前期抽帧而漏掉 Word、PDF、QA：

```bash
python scripts/lapian_finalize_manifest.py --project-dir 08拉片输出/YYYYMMDD/片名_导演级拉片
```

## 音频分析工作流

### 0. 推荐自动准备方式

如需完整证据包，优先使用：

```bash
python scripts/prepare_lapian_evidence.py path/to/video.mp4 --out-root 08拉片输出 --project-name 片名 --task-name 导演级拉片 --interval 1
```

该脚本会自动建立标准目录、记录 `ffprobe` 摘要、调用逐秒抽帧、提取 16kHz 单声道 WAV、生成每秒音量 CSV，并写入根目录 `manifest.json`。

ASR 默认不自动运行。只有明确需要对白转写时再加：

```bash
python scripts/prepare_lapian_evidence.py path/to/video.mp4 --project-name 片名 --interval 1 --asr-model base
```

如果 ASR 超过 3 分钟没有明确进展，遵守项目规则：停止反复等待，改为检查已落盘文件、日志、模型缓存和最小健康检查。

### 1. 音轨检查

先记录：

- 是否有音轨。
- 编码、采样率、声道、总时长。
- 音频与视频时长是否基本一致。
- 是否存在字幕、对白稿、ASR 文件或用户提供文本。

### 2. 音频提取

优先提取为 16kHz 单声道 WAV，供 ASR 和能量分析使用。输出文件放入：

```text
<输出目录>/<片名>_音频分析/
```

推荐保留：

- `<片名>_audio_16k_mono.wav`
- `<片名>_audio_levels.csv`
- `<片名>_asr_*.json`
- `<片名>_asr_*.txt`
- `<片名>_asr_*.jsonl`（如果工具支持逐段输出）

### 3. ASR 与对白规则

- 有官方字幕或用户提供字幕时，字幕优先；ASR 用于校验、补漏、定位声音节奏。
- 无字幕时，ASR 可作为对白初稿，但必须标注“ASR 可能误识别”。
- 对白分析不是单纯翻译台词，而要写出“表面意思、实际行动、潜台词、关系变化、叙事功能”。
- 听不清、噪声遮挡、多人重叠时，不要硬编台词；标为“听辨不可靠 / 需人工复听”。

### 4. 音乐、音效与声音结构

每个主要段落至少判断：

- 音乐：进入点、退出点、情绪、速度、配器倾向、是否推动节奏或反讽画面。
- 环境声：空间感、距离感、城市/自然/机械/室内/人群质感。
- 音效：动作音、转场音、低频冲击、电子声、门/脚步/呼吸/衣料等可辨识声音。
- 人声：对白、旁白、喘息、笑声、哭声、群体声、电子处理声。
- 静默：是否形成压力、悬念、情绪断点或信息空白。
- 声画关系：同步、错位、先声夺人、声音桥、声画反差。

如果没有专业分轨，禁止断言“某一条音轨的真实混音设计”。可以说“成片听感上呈现为……”。

## 逐秒图文拉片写作规则

### 1. 证据留档，报告 0.5-2 秒自适应展示

图文拉片分为两层：证据层默认 1 秒抽帧并完整留档，必要时对高密度局部补 0.5 秒证据；报告展示层根据内容密度在 0.5-2 秒之间自适应选择展示帧。不要把“每秒证据”误写成“每秒都必须印在报告或 Word 里”。

自适应展示策略：
- 常规段落：约 2 秒一张图。
- 关键变化：字幕/对白变化、音频峰值、硬切、人物表情转折、动作爆点、重要道具/人物首次出现，回到 1 秒展示。
- 高密度短段：喜剧节奏、打斗、快速剪辑、表情细节、强动作或快速变形，可用 0.5 秒展示。
- 重复段落：静态凝视、片尾名单、长时间同构图移动、情绪延宕，只保留首/中/尾代表帧，并用文字说明连续性。
- Word / 专业导演分析 / 风格圣经：默认只嵌入被选中的展示帧、专业分析代表图、段落关键图和资产卡图片，不嵌入每秒全量证据帧。

优先生成报告展示帧清单：

```bash
python scripts/select_lapian_report_frames.py --manifest 01_逐秒抽帧/frames_manifest.json --subtitle-json 02_音频分析/片名_asr_from_vtt_subtitles.json --levels-csv 02_音频分析/片名_audio_levels.csv --report-interval 2 --out 03_MD报告/片名_报告展示帧清单.json --csv-out 03_MD报告/片名_报告展示帧清单.csv
```

字幕和音量 CSV 是可选输入，缺失时清单仍可生成。约 1 分钟以内的短片把 `--top-audio-peaks` 降到 3-5，否则默认 20 个音频峰值会让几乎每一秒都成为关键秒，稀疏展示失效。

如果用户明确要求“每一秒都要放图/逐秒全部图”，才在报告里打印全部 1fps 图片行；否则按 0.5-2 秒自适应展示帧清单写报告。

每个段落建议包含：

1. 段落功能：这一段让观众知道什么、感到什么、期待什么。
2. 叙事变化：人物目标、阻力、信息、价值状态如何变化。
3. 视听语言：景别、构图、视角、运动感、光色、美术、剪辑、声音。
4. 表演情绪：表情、眼神、身体姿态、行动速度、情绪层次。
5. 可学方法：这段可迁移到剧本、分镜、AI 视频提示词里的设计原则。

### 2. 逐秒表要轮换观察轴

逐秒表中的“变化”列应轮换以下观察轴，避免连续十几行都是同一句式：

- 叙事信息变化。
- 视线和注意力引导。
- 主角表情/身体状态。
- 镜头距离与景别变化。
- 构图重心和前中后景关系。
- 光色、道具、场景信息。
- 声音进入、退出或强弱变化。
- 剪辑点、转场、节奏密度。
- 主题意象或符号被强化/反转。

### 3. 镜头运动与剪辑点判断要分级

剪辑点优先使用 `01_逐秒抽帧/片名_镜头切分.json`（`prepare_lapian_evidence.py` 默认生成，或单独运行 `detect_shot_cuts.py`）。有切分清单时，镜头表、镜头数、平均/中位镜头时长直接引用清单数据，标注为“基于场景检测的镜头切分”；硬切可视为接近 E1 的证据，但叠化、慢转场可能漏切，快速运动、闪烁可能误切，存疑剪辑点要对照抽帧复核。没有清单时才退回“粗略镜头切分”。

仅凭 1 秒抽帧不能精确确认推拉摇移跟升降。写法必须分级：

- E1：连续帧或视频片段直接显示运动。
- E2：相邻帧尺寸、透视、主体位置变化支持“运动感”判断。
- X：无法确认真实运镜，只能描述构图变化。

可用表达：

- “从抽帧看呈现轻微推进感，需视频连续播放确认真实运动。”
- “这一秒的画面结果像是横移或跟拍，但证据只来自相邻帧位置变化。”
- “当前证据只能确认景别变化，不能确认是否为镜头运动或剪辑造成。”

## 飞书与 Word 嵌图交付规则

### 1. 格式选择

| 交付方式 | 推荐度 | 原因 |
|---|---:|---|
| Markdown + 本地图片路径 | 中 | 适合本机阅读和二次编辑，但上传飞书时图片可能断链。 |
| Markdown 内嵌 base64 图片 | 低 | 文件巨大，很多平台不稳定，编辑体验差。 |
| DOCX 嵌图版 | 高 | 图片真正嵌入，适合飞书、Word、WPS 和客户交付。 |
| PDF 预览版 | 中 | 适合审阅效果，但不适合继续编辑。 |

用户说“上传飞书直接上传 MD”时，应说明：MD 的图片多半只是路径引用；为了飞书稳定显示，优先转 DOCX 嵌图版。

### 2. DOCX 制作要求

- 优先使用 `scripts/md_image_report_to_docx.py` 生成嵌图版 DOCX；如果脚本不适合当前报告结构，再用 Documents 插件或 python-docx 自定义转换。
- 保留原 Markdown，不覆盖。
- 生成文件名使用中文语义后缀，例如：`片名_导演级视听逐秒拉片报告_嵌图版.docx`。
- 图片过大时先压缩生成缓存目录，但不要删除原抽帧。
- DOCX 中每一张报告图片都要以文档媒体形式嵌入。
- 专业导演分析 DOCX 同样必须真实嵌图。默认使用每个大场景 3-8 张代表图，重点让文字解释导演意图和观众心理，不重复主拉片报告的完整帧表。
- 源片风格圣经 DOCX 同样必须真实嵌图。默认只嵌入段落关键图和资产卡图片，不嵌入逐秒全量抽帧，以控制在 20-50 页左右。
- 表格列宽、字号和图片宽度要适合阅读，不让图片或长文字撑坏页面。

推荐命令：

```bash
python scripts/md_image_report_to_docx.py path/to/report.md --out path/to/report_嵌图版.docx --compress
```

### 3. DOCX 验证要求

生成后至少检查：

- Markdown 中图片引用数量。
- DOCX 内 `word/media/` 图片数量。
- DOCX 内绘图元素 / `a:blip` 引用数量。
- 是否存在缺失图片路径。
- 如果可用，导出 PDF 或渲染预览页，抽查是否有空白页、断图、表格严重错位。

如果 Word COM、LibreOffice、Pandoc 或 PDF 渲染工具不可用，最终回复必须说明验证缺口。

## 最终质量审计

交付前先完成 SKILL.md 规定的“批判复审 pass”（以挑刺立场重读草稿，处理过度解读、无证据运镜、反证遗漏、套话复读、注意力平均化），修订后再进入证据完整性审计。

交付前做一次“证据完整性审计”，至少确认：

- 视频时长、抽帧间隔、抽帧数量与最后一秒是否对应。
- Markdown 图片引用数量是否等于应有报告展示帧数量；如果用户明确要求全量逐秒图，才等于完整抽帧数量。
- 第一秒和最后一秒是否都在报告里。
- 缺失图片引用数量是否为 0。
- 如用户要求音频：ASR / 字幕 / 音频能量文件是否存在，声音分析是否写进正文。
- 报告是否包含：叙事结构、视听语言、主角表情情感、主题、声音、对白/旁白、可学习方法。
- 完整本地视频拉片是否先生成或引用了“帧/自适应详细拉片报告”作为基础证据。
- 完整本地视频拉片是否额外生成独立专业导演分析；专业导演分析是否包含导演意图层、观众心理层、视听执行层、主题表达层和可学习方法。
- 完整本地视频拉片是否额外生成独立源片风格圣经；风格圣经是否包含段落式图片资产索引、场景级资产卡、元素级资产卡、图像变量模板、视频变量模板。
- DOCX 是否含真实嵌图引用，PDF 是否存在且具备 `%PDF-` 文件头和 `%%EOF` 结束标记。
- `manifest.json` 是否记录了最终 Markdown、DOCX、PDF、QA、ASR 和抽帧路径。
- 中文文件是否为 UTF-8；如果 PowerShell 显示乱码，用 Python codepoint 或编辑器实际打开结果确认，不要只相信终端显示。

## 图文报告滚动展示视频交付规则

完整本地视频拉片任务完成主 Markdown 图文报告后，必须额外生成一个展示视频，除非用户明确说不要。

默认规则：
- 输出目录：`07_视频展示/`。
- 视频布局：竖屏 `1080x1920`；上半屏播放源视频并保留原声；下半屏滚动展示主 Markdown 图文拉片报告。
- 下半屏必须是“带图片的报告滚动”，不是纯文字摘录；必须保留抽帧缩略图、时间戳、节点记录、字幕/声音说明和段落标题。
- 默认使用主拉片报告作为滚动来源，不使用专业导演分析或风格圣经，除非用户特别指定。
- 文件命名：`片名_上视频下图文报告滚动展示_v01.mp4`，同名存在时递增版本，不覆盖旧文件。
- 如果 Markdown 报告后续改动，展示视频视为过期，需要重新生成。

推荐命令：
```bash
python scripts/make_lapian_showcase_video.py --project-dir 08拉片输出/YYYYMMDD/片名_导演级拉片
```

如果 manifest 里的源视频路径不可用：
```bash
python scripts/make_lapian_showcase_video.py --project-dir 08拉片输出/YYYYMMDD/片名_导演级拉片 --video path/to/source.mp4
```

最终审计时额外确认：
- `07_视频展示/` 下存在展示 MP4。
- 用早/中/晚至少 3 张预览帧检查，下半屏确实有图片和时间戳。
- `ffprobe` 显示成片时长与源视频时长基本一致，并且有音频流。

最终回复路径列表中必须包含：
- `07_视频展示/片名_上视频下图文报告滚动展示_v01.mp4`
